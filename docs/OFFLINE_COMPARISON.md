# 오프라인 대비 자료 — 온바디 경로 vs 기존 앱 (Story 2.3, AC3)

작성 2026-07-22 · 근거: FR3(오프라인 우선), P-1(43.4% — "서버 죽으면 아무것도
안 됨"), NFR6(모르는 것은 수치화하지 않는다)

> [!warning] 이 표는 **틀**이지 완성된 결과가 아니다
> AC3("기존 ThinQ 앱 경로를 동일 조건에서 시도")은 **코드로 완결되지 않는다.**
> 실제 앱을 기내모드에서 돌리는 것은 사람이 폰으로 하는 일이다.
> 우리 경로는 실측으로 채우고, **ThinQ 앱 쪽은 "미측정"으로 남긴다.**
> 추정치를 적지 않는다 — 발표에서 반박당하는 지점이 정확히 여기다.

---

## 1. 대비표

| 경로 | 네트워크 차단 시 제어 | 재등록 필요 | 데이터 소재 | 측정 상태 |
|---|---|---|---|---|
| **온바디 프로필 (본 프로젝트)** | ✅ 성공 (차단 상태 종단 테스트) | 불필요 | 온바디 | 실측 — 아래 §2 |
| **ThinQ 앱 (기존)** | 미측정 | 미측정 | 미측정 | **미측정** |

우리 경로의 "차단 시 성공"은 `tests/test_offline_enforcement.py::
test_offline_result_identical_to_online`가 고정한다 — 연결/차단 두 상태의
**최종 상태·이벤트 로그가 동일**함을 단언한다("둘 다 성공"보다 강한 단언).

## 2. 우리 경로 측정 (실측)

| 항목 | 값 | 출처 |
|---|---|---|
| 차단 상태 루틴 실행 | 오류 0건 완료 | `execute_routine` under `enforce_offline()` |
| 재조립 위치 | 수신 측 (`reassembled_by="receiver"`) | R5 해소 후 |
| 최종 상태 == 연결 상태 | 바이트 단위 동일 | 종단 동등성 테스트 |
| 네트워크 접근 시도 | 0건 (시도 시 `OfflineViolation`) | 하네스 |

⚠️ 이 측정의 한계는 `docs/OFFLINE_EVIDENCE.md §2`에 있다 —
프로세스 내 차단이지 장비 차단이 아니다.

## 3. ThinQ 앱 측정 절차 (미실행 — 틀만 정의)

앱 측정을 하게 되면 **아래를 반드시 함께 기록**한다. 기록 없는 비교는
발표에서 반박당한다:

- [ ] **앱 버전** (설정 → 정보)
- [ ] **기기** (모델·OS 버전)
- [ ] **측정 일자**
- [ ] **캡처** (기내모드 화면 + 앱 반응 화면)
- [ ] 시나리오: 기내모드 ON → 등록된 가전 제어 시도 → 결과
- [ ] 시나리오: 앱 삭제 → 재설치 → 프로필 잔존 여부

**현재 상태: 미측정.** 이 절이 실측으로 채워지기 전까지 §1의 ThinQ 행은
"미측정"으로 남는다. `test_comparison_doc_leaves_thinq_unmeasured`가 이를 지킨다.

## 4. 근거 연결

P-1(43.4%)의 대표 인용이 이 대비의 존재 이유다 — VOC가 "서버·앱이 죽으면
아무것도 안 됨"을 말했고, 이 표가 **온바디 구조가 그 문제를 없앤다**는 것을
보인다. 단, 그 주장은 ThinQ 칸이 실측으로 채워질 때 완성된다.
그 전까지는 "우리 경로는 차단에서 동작한다"까지만 말한다(NFR6).
