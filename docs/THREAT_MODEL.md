# 위협 모델 — 릴레이 공격 (Story 4.3, NFR1)

> **AI 초안 — 최종 책임은 팀.** 이 문서는 근접 명령의 릴레이/재생 위협과 방어,
> 그리고 **잔여 한계**를 기록한다(AC2). 코드 원천은 `home_profile/proximity.py`의
> `ProximityGuard`이며, `tests/test_relay_defense.py`가 문서-코드 정합을 회귀 고정한다.

## 위협

온바디 프로필이 로컬(BLE)에서 가전을 여는 구조는 **근접성이 인증**이 된다 —
"손목에 있으면 열린다". 그래서 두 공격이 표면이 된다:

- **재생(replay)**: 공격자가 유효한 근접 명령을 캡처해 **나중에** 재사용.
- **중계(relay)**: 공격자가 근접 명령을 **실시간으로** 원거리로 중계 — 워치는
  집에, 공격자는 문 앞에서 마치 워치가 근처인 것처럼 명령을 성립시킨다.

## 방어 — 챌린지-응답 신선도

가전이 명령마다 **신선한 nonce 챌린지**를 발급하고, 근접한 워치만 그 nonce를
(신선도 창 안에) 받아 응답할 수 있다. `ProximityGuard.verify`는 토큰 nonce가
**현재 챌린지와 일치 + 미소비**일 때만 통과시키고, 통과한 nonce는 **1회용**으로
소비한다.

이 방어는 "손목에 있으면 열린다"를 **"손목에 있고 + 신선한 챌린지에 실시간
응답해야 열린다"**로 좁힌다. 완전 제거가 아니라 **표면 축소**다.

## 방어 범위 — 막는 것 / 못 막는 것

| | 시나리오 | 결과 |
|---|---|---|
| 🛡️ **막는다** | 캡처한 명령의 재생(replay) | 소비된 nonce → 거부 |
| 🛡️ **막는다** | 옛/위조 nonce로 만든 명령 | 현재 챌린지 불일치 → 거부 |
| 🛡️ **막는다** | 같은 챌린지에 두 번 응답 | 1회용 소비 → 두 번째 거부 |
| ⚠️ **못 막는다** | **실시간 relay** — 신선도 창 안에서 챌린지·응답을 실시간 중계 | 신선도만으로는 구분 불가 |
| ⚠️ **못 막는다** | 물리적 근접 위조(공격자가 실제로 근처) | 근접이 곧 인증이므로 성립 |
| ⚠️ **못 막는다** | 손목 탈취 후 즉시 사용 | 4.4 폐기(revocation)의 영역 |

## ⚠️ 잔여 한계 (완전 방어를 주장하지 않는다)

**우리는 릴레이를 완전히 막지 못한다.** 신선도(nonce)는 **재생**을 막지만,
**실시간 relay**는 막지 못한다 — 공격자가 신선한 챌린지를 원거리 워치로 실시간
전달하고 응답을 되받아 신선도 창 안에 제출하면, 신선도 검사만으로는 근접한
응답과 구분할 수 없다.

진짜 relay를 막으려면 **거리 바운딩(distance bounding)** — 챌린지-응답의 왕복
시간(RTT)을 물리적으로 측정해 응답자가 광속 한계 안의 거리에 있는지 확인 — 이
필요하다. 이는 나노초급 타이밍 하드웨어를 요구하며 **이 시뮬레이터에는 없다**.
따라서 실시간 relay 방어는 **범위 밖**이며, 실기기 단계의 과제로 남긴다.

이 정직한 경계 설정이 3.1의 오프라인 한계 표기("프로세스 차단이지 장비 차단
아님")와 같은 규율이다 — **막지 못하는 것을 막았다고 말하지 않는다**(NFR6).

## 발표에서 말해도 되는 것 / 안 되는 것

| 말해도 되는 것 | 말하면 안 되는 것 |
|---|---|
| "캡처한 명령의 재생을 신선도로 막는다" | ❌ "릴레이 공격을 완전히 막았다" |
| "'손목에 있으면 열린다'를 '신선한 챌린지에 실시간 응답해야'로 좁혔다" | ❌ "원격 공격이 불가능하다" |
| "실시간 relay는 거리 바운딩이 필요해 범위 밖" | ❌ "우리 로컬 통신은 안전하다"(무조건형) |
