# 데이터 소재 — "내 집 정보는 어디에 있는가" (Story 4.2, FR7/AC1·AC2)

> **AI 초안 — 최종 책임은 팀.** 이 문서는 프로필 원본의 위치와 서버 미보유 사실을
> 명시한다(AC1). 코드 원천은 `home_profile/residency.py`의 `data_residency`이며,
> `tests/test_data_residency.py`가 문서-코드 정합을 회귀로 고정한다.

P-3 반박 마무리(6.2%). 4.1(무계정 온보딩)이 "신원을 안 넘긴다"였다면, 4.2는
**"그럼 데이터는 어디"**에 답한다. 서버가 원본을 갖지 않는 것은 결함이 아니라
**프라이버시 셀링포인트**다. "서버에 없다"를 말이 아니라 **관찰**로 증명한다.

## 데이터 소재 요약

| 항목 | 값 | 증거 |
|---|---|---|
| 프로필 원본 위치 | **온바디** (참조 어댑터 레코드 — 실기기 아님) | `data_residency.profile_location` |
| 서버 원본 보유 | **없음** | `server_holds_original = False` |
| 서버로 전송되는 항목 | **없음** | `server_transmitted = []` (AC2: 없으면 빈 목록) |
| 온바디만으로 복원 | **가능** | `restore_from_carrier` == 원본 (원본이 거기 있다는 증거) |

## "서버에 없다"의 증명 방식 (말이 아니라 관찰)

1. **온바디 footprint**: 온바디에 실제 무엇이 얼마나 있는지 — 레코드 종류(meta·
   기기·루틴)별 개수와 바이트. ⚠️ 이름(device_ref)은 원문으로 노출하지 않는다 —
   사용자 통제 토큰일 수 있어 개수·크기로만 보고한다.
2. **온바디 복원 증거**: 캐리어(온바디)만으로 프로필이 완전히 복원되어 원본과
   동일하다(`restorable_from_onbody = True`). 다른 저장소(서버)가 필요 없음을
   관찰로 보인다 — 원본이 서버가 아니라 온바디에 완결돼 있다.
3. **네트워크 0**: 소재를 알아내는 데 서버를 부르지 않는다("서버에 없다는 걸 서버에
   물어봤다"는 자기모순 금지). `enforce_offline` 강제 안에서 소재 확인이 성공함으로
   증명(`test_residency_under_offline_enforcement`).

## FR7 계보 (여러 스토리가 모여 "서버 원본 미보유"를 이룬다)

- **1.1 스키마**: 식별자·계정·위치 키를 거부(FORBIDDEN_KEY_FRAGMENTS) — 애초에
  서버에 보낼 신원 데이터가 프로필에 없다.
- **3.1 복원**: 재설치 후 복원에 클라우드 조회 0회 — 원본이 서버에 없어도 복원된다.
- **4.1 온보딩**: 계정·로그인 없이 온보딩 완결 — 서버에 계정 데이터를 만들지 않는다.
- **4.2 소재(이 문서)**: 원본이 온바디에 있고 서버 전송 항목이 '없음'임을 명시.

## ⚠️ 한계 (정직 표기, NFR6)

이 증명은 **이 참조 어댑터·이 파이썬 프로세스에서 서버로 가는 경로가 없음**까지다.
- 실제 벤더 워치(가민 등)·실가전 연동 시의 데이터 흐름은 **범위 밖**이다(시뮬레이터 기반).
- "서버에 없다"는 우리 코드 범위의 관찰이지, 벤더 SDK나 OS 레벨 텔레메트리까지
  보증하는 주장이 아니다(3.1 오프라인 한계 표기와 같은 규율 — 과장하지 않는다).

## 심사위원 예상 질문

**Q: "서버에 없다는 걸 어떻게 믿죠?"**
A: 온바디만으로 프로필이 복원됩니다(화면에서 확인). 다른 저장소가 필요 없다는
관찰이며, 소재 확인 자체가 오프라인 강제 안에서 성공합니다. 다만 범위는 위 '한계'
절대로 우리 코드까지입니다.

**Q: "그럼 클라우드 백업은 아예 없나요?"**
A: 이 구조에는 없습니다(`server_transmitted = 없음`). 백업은 온바디→온바디(3.1
재설치 복원)로 해결하며, 서버 백업을 셀링포인트를 깨면서 도입하지 않습니다.
