# SPOF 없음 — 증거와 그 한계 (Story 2.2)

작성 2026-07-22 · 근거: FR2(BLE 로컬 통신만으로 제어), FR3(오프라인 우선),
NFR6(근거 무결성 — 모르는 것은 수치화하지 않는다)

> **용어**: 우리가 없앤 것은 **SPOF(Single Point of Failure, 단일 장애점)**다 —
> 클라우드 서버 하나가 죽으면 집 전체가 멈추는 구조(P-1, 1점 리뷰 30.0%). LG
> 자기언어로는 **Effortless의 위반**이다(`LGE_KEYWORD_MATRIX.md`). "서버 없이"는
> 대중어이고, 개발자 심사위원에겐 **SPOF 제거**가 정확한 용어다.

이 문서는 **"클라우드 호출이 하나도 없다"(AC2)를 무엇으로 증명했고,
그 증명이 어디까지만 유효한지**를 적는다. 한계를 적지 않은 증거는
이 프로젝트에서 증거가 아니다.

---

## 1. 확보한 증거 2종

### 1-A. 동적 증거 — 실행 중 네트워크 호출 감시

`tests/test_routine_execution.py::test_no_network_calls_during_execution`

루틴 실행 경로 전체(캐리어 읽기 → 역직렬화 → 루틴 변환 → 청킹 → 재조립 →
전송 → 상태 반영)를 도는 동안, 호출되면 즉시 실패하는 가짜를 심어둔다:

| 감시 대상 | 막는 것 |
|---|---|
| `socket.socket` | 파이썬에서 소켓을 여는 **가장 흔한** 경로 |
| `socket.create_connection` | 고수준 연결 헬퍼 |
| `urllib.request.urlopen` | 표준 라이브러리 HTTP |
| `subprocess.Popen` / `subprocess.run` | 외부 프로세스(`curl` 등)를 통한 우회 |

**주장 가능한 것**: 이 경로는 위 API를 한 번도 부르지 않는다.

> ⚠️ **"모든 연결을 막는다"고 쓰지 않는다.** 초판 문구는 `socket.socket`을
> "모든 TCP/UDP 연결의 출발점"이라 적었는데 그것은 과장이다 —
> `socket.socketpair()`·`socket.getaddrinfo()`(DNS)·C 확장이 직접 부르는
> syscall은 이 감시를 지나간다. **한계를 적는 문서가 자기 앞장에서 과장하면
> 그 문서의 신뢰가 먼저 죽는다**(리뷰 2026-07-22, Paige).

### 1-B. 정적 증거 — 네트워크 라이브러리 import 부재

`tests/test_routine_execution.py::test_no_network_imports_static`

실행 경로 7개 모듈(`home_profile/` 4 + `appliance_sim/` 3)의 AST를 파싱해
네트워크 토큰 import가 없음을 단언한다:
`socket`·`urllib`·`http`·`requests`·`httpx`·`aiohttp`·`ftplib`·`telnetlib`·
`smtplib`·**`subprocess`**(외부 프로세스 우회 차단 — 리뷰에서 추가).

Story 1.3 리뷰(P3)에서 강화한 검사기를 재사용한다 —
**동적 import(`importlib.import_module`·`__import__`)도 탐지**하고,
주석·docstring의 언급은 위반으로 치지 않으며(AST 기반),
검사기 자신이 가짜 소스에서 실패함을 별도 테스트로 고정한다
(`test_network_checker_detects_injected_import` — 아무것도 안 하는 검사기도
"위반 없음"을 내기 때문).

**주장 가능한 것**: 이 모듈들에는 네트워크로 나가는 코드 경로가 존재하지 않는다.

## 2. ⚠️ 이 증거의 한계 — 발표에서 말할 수 있는 것과 없는 것

**파이썬 레벨 감시는 프로세스 밖을 못 본다.** 구체적으로 다음을 보지 못한다:

- OS 레벨 통신 (네이티브 확장, 서브프로세스, 드라이버)
- 워치 펌웨어·Connect IQ 런타임의 동작
- 우리가 감시 목록에 넣지 않은 서드파티 경로
- 실제 무선 구간에서 오가는 패킷

**차단(2.3)의 추가 한계 — 모듈 속성만 막는다** (리뷰 2026-07-22, Paige):
`offline_guard`는 `socket.socket`이라는 **모듈 속성**을 blocker로 갈아끼운다.
그래서 `socket.socket()`처럼 매번 모듈을 거쳐 부르는 코드는 막히지만,
`from socket import socket`으로 **import 시점에 지역 이름으로 바인딩**해둔
코드는 그 지역 참조를 쓰므로 **막히지 않는다.** 이것은 2.2 AST 한계와 같은
계열이다 — 우리 코드(`home_profile/`·`appliance_sim/`)는 전부 `socket.socket()`
형태라 현재는 새지 않지만(grep 확인), 서드파티가 사전 바인딩하면 통과한다.
그러므로 차단은 **"부를 수 없다"가 아니라 "모듈 속성으로 부르면 못 부른다"**이다.

따라서:

| 말해도 되는 것 | 말하면 안 되는 것 |
|---|---|
| "우리 코드에 네트워크 경로가 없음을 테스트로 고정했다" | ❌ "네트워크 캡처로 확인했다" |
| "실행 중 소켓·HTTP 호출이 0건임을 회귀 테스트가 지킨다" | ❌ "패킷 수준에서 검증했다" |
| "정적·동적 두 각도의 증거가 있다" | ❌ "서버 통신이 없음을 증명했다"(무조건형) |

**Story 2.3에서 한 걸음 나아갔다** — `offline_guard.enforce_offline()`가
프로세스 내 네트워크를 **실제로 막고**(감시가 아니라 차단), 그 상태에서
루틴 실행이 연결 상태와 **동일한 결과**를 냄을 종단 테스트로 고정했다
(`tests/test_offline_enforcement.py`). 2.2의 "부르지 않았다"가 2.3에서
"부를 수 없다"가 됐다.

단 **여전히 프로세스 내 차단이다.** 장비 차단(기내모드)은 사람이 누르는 것이고
코드가 관측하지 못한다 — 발표 시연 절차는 `docs/DEMO_SCRIPT.md §3`에 있고,
ThinQ 앱 대비는 `docs/OFFLINE_COMPARISON.md`에서 **미측정**으로 남아 있다.
위 "말해도 되는 것 / 안 되는 것" 표는 2.3 이후에도 그대로 유효하다.

## 3. 왜 이렇게까지 하는가

FR7("프로필 원본이 서버에 존재하지 않으며 데이터 소재를 사용자에게 명시")과
P-3(6.2%) 인용 — *"내 집에 있는 제품들 연결해서 쓰는 건데 굳이 회원가입을
강요하는 이유가 뭡니까"* — 이 프로젝트의 셀링포인트는 **서버가 원본을 갖지 않는
구조**다. 그 주장을 코드가 뒷받침하지 못하면 발표의 한 축이 무너진다.

동시에, 증거를 실제보다 강하게 말하면 **같은 축이 다른 방향으로 무너진다.**
그래서 증거와 한계를 같은 문서에 적는다. 이것이 NFR6의 실천이다 —
'산출 불가' > 그럴듯한 숫자.
